# AGENTS.md

## 프로젝트

이 저장소는 `codex-harness-tgr-v1`입니다.

목표는 이전 v0 프로젝트에서 검증한 최소 Codex 하네스를 두 번째 Unity 프로젝트에 이식하고, Unity 프로젝트를 반복 가능하게 설계, 데이터화, 구현, 검증하는 v1 하네스를 검증하는 것입니다.

## 먼저 읽기

파일을 변경하기 전에 다음을 읽습니다.

1. `README.md`
2. `AGENTS.md`
3. `docs/current-state.md`가 있으면 읽습니다.
4. `docs/decisions.md`가 있으면 읽습니다.
5. 작업에 ExecPlan이 필요하면 `PLANS.md`를 읽습니다.
6. `exec-plans/` 아래의 관련 ExecPlan이 있으면 읽습니다.

## 규칙

- ExecPlan에서 요청하지 않는 한 게임플레이를 구현하지 않습니다.
- 아직 MCP, 사용자 정의 skill, hook, subagent, 외부 패키지를 추가하지 않습니다.
- 변경은 작고 검증 가능하게 유지합니다.
- 모든 저장소 문서는 한글로 작성합니다.
- 작업으로 프로젝트 상태가 바뀌면 프로젝트 문서를 업데이트합니다.
- 저장소 파일에 개인의 경력, 가족, 재정, 고용주 관련 민감 정보를 포함하지 않습니다.

## Design Baseline Check

Player-facing gameplay 설계, gameplay ExecPlan 작성, gameplay 구현을 시작하기 전에 Codex는 다음을 확인합니다.

1. `docs/design/README.md`가 존재한다.
2. `docs/design/core-beliefs.md`가 존재한다.
3. `docs/design/core-beliefs.md`의 상태가 정확히 `상태: 승인됨`이다.

`docs/design/README.md`는 design 문서 작성 규약이므로 승인 상태 확인 대상이 아니라 존재 확인 대상입니다.

`docs/design/core-beliefs.md`가 없거나 `상태: 작성 전`, `상태: 초안`, `상태: 수정 필요`처럼 `상태: 승인됨`이 아니면 Codex는 gameplay 설계, gameplay ExecPlan 작성, gameplay 구현을 멈추고, 사용자에게 design baseline 생성 또는 승인이 먼저 필요하다고 알립니다.

`상태: 작성 전`인 `core-beliefs.md`는 파일이 존재해도 design 산출물이 아닙니다. 사용자 인터뷰와 공유된 이해 검증을 거쳐 `상태: 초안`으로 다시 작성되고, 사용자가 명시 승인해 `상태: 승인됨`이 되기 전까지 gameplay 기준으로 사용할 수 없습니다.

하네스, 문서 구조, workflow 규칙 도입 작업에는 이 체크를 적용하지 않습니다. 단, 하네스 또는 문서 작업 중 player-facing gameplay 규칙을 정의하기 시작하면 Design Gate로 전환합니다.

Codex는 사용자 명시 승인 없이 `docs/design/core-beliefs.md` 또는 다른 design 문서의 상태를 `상태: 승인됨`으로 변경하지 않습니다.

## 요청 분류

Codex는 사용자 요청을 먼저 분류합니다.

- 분석, 논의, 검토 요청이면 파일 변경, ExecPlan 생성, 구현을 하지 않습니다.
- 오타 수정, 작은 서식 수정, 작은 문서 문구 보정처럼 사용자가 명확히 요청한 작은 변경은 ExecPlan 없이 처리할 수 있습니다.
- 게임플레이, Unity 씬, 워크플로우, 구조 변경은 사소하지 않은 작업으로 보고 ExecPlan 흐름을 따릅니다.

## 사소하지 않은 작업 흐름

사소하지 않은 작업은 바로 ExecPlan을 만들거나 구현하지 않습니다.

1. 논의한다.
2. `grill-me` 방식 인터뷰를 진행한다.
3. 공유된 이해를 검증한다.
4. 사용자에게 ExecPlan 생성 여부를 확인한다.
5. ExecPlan을 생성한다.
6. Codex가 ExecPlan을 자체 리뷰한다.
7. 사용자 검토용 요약을 제공한다.
8. 사용자에게 구현 승인 여부를 확인한다.
9. 구현한다.
10. 검증하고 문서를 갱신한다.

인터뷰 규칙:

- 질문은 한 번에 하나만 합니다.
- 각 질문에는 Codex의 추천 답안을 포함합니다.
- 코드나 문서에서 확인할 수 있는 사실은 사용자에게 묻지 않고 직접 확인합니다.
- 사용자의 요청이 충분히 구체적이면 질문 수를 줄이고, 되돌리기 비싼 결정이나 남은 모호성만 확인합니다.

공유된 이해 검증에서는 목표, 성공 기준, 포함 범위, 제외 범위, 제약, 검증 방법, 위험한 가정, 남은 열린 질문, 되돌리기 비싼 결정을 요약하고 사용자에게 의도와 맞는지 확인합니다.

공유된 이해 검증이 끝난 뒤에만 다음 질문으로 ExecPlan 생성 승인을 확인합니다.

```text
이 내용으로 ExecPlan을 만들까요?
```

이 질문 직후의 짧은 긍정 답변은 ExecPlan 생성 승인으로 봅니다. 공유된 이해 검증 없이 나온 일반적인 동의 표현은 ExecPlan 생성 승인으로 보지 않습니다.

새 ExecPlan은 기본적으로 `구현 승인 대기` 상태로 시작합니다. ExecPlan 생성 후 Codex는 인터뷰 합의와 계획 내용이 일치하는지, 목표와 범위와 검증 방법이 빠지지 않았는지, 구현 승인 전 구현 작업이 포함되지 않았는지 자체 리뷰합니다.

자체 리뷰 후에는 목표, 포함 범위, 제외 범위, 검증 방법, 현재 상태, 사용자가 특히 확인해야 할 결정 3개 이하를 요약합니다.

그 뒤에만 다음 질문으로 구현 승인을 확인합니다.

```text
이 ExecPlan대로 구현을 시작할까요?
```

이 질문 직후의 짧은 긍정 답변은 구현 승인으로 봅니다. 검토 중 나온 일반적인 동의 표현은 구현 승인으로 보지 않습니다.

별도 `game-brief` 또는 `game-spec` 문서는 기본 산출물로 만들지 않습니다. 검토용 HTML이나 ExecPlan 요약 문서 같은 보조 산출물은 계획이 복잡하거나 사용자가 요청하고 동의한 경우에만 생성합니다.

## 문서 갱신

현재 단계, 프로젝트 상태, 활성 계획, 다음 단계, 알려진 제한 사항이 바뀌면 `docs/current-state.md`를 업데이트합니다.

오래 유지해야 하는 프로젝트 결정이 생기면 `docs/decisions.md`를 업데이트합니다.

오래 유지해야 하는 결정에는 다음이 포함됩니다.

- 이름 규칙
- 폴더 구조
- 워크플로우 규칙
- 도구 채택 또는 보류
- 아키텍처 방향
- 검증 전략
- 단계별 범위

일시적인 채팅 내용, 개인의 경력, 가족, 재정, 고용주 관련 민감 정보는 저장소 문서에 기록하지 않습니다.

## 완료 기준

작업은 다음을 모두 만족해야 완료입니다.

1. 요청한 변경이 존재합니다.
2. 검증 절차가 기록되어 있습니다.
3. 활성 plan 또는 프로젝트 상태가 업데이트되어 있습니다.
4. commit 후 `git status`가 clean입니다.

## 실행 계획 (ExecPlan)

사소하지 않은 작업은 설계부터 구현까지 `PLANS.md`에 정의된 ExecPlan을 사용합니다.

ExecPlan은 저장소 루트의 `exec-plans/` 아래에 생성하고 유지합니다.

관련 ExecPlan이 존재하거나 갱신되기 전에는 구현을 시작하지 않습니다.

사소하지 않은 작업에서는 공유된 이해 검증과 ExecPlan 생성 승인 없이 새 ExecPlan을 만들지 않습니다. 생성된 ExecPlan은 자체 리뷰와 사용자 검토용 요약, 구현 승인 확인을 거친 뒤에만 구현에 사용할 수 있습니다.
