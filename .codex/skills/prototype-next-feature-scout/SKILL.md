# Prototype Next Feature Scout

## 목표

작은 게임 프로토타입에서 다음으로 만들 "가치 있는 하나의 추가"를 고르라. 코드 품질 개선, 리팩터, 아키텍처 정리, 장기 로드맵이 아니라 플레이어가 바로 느끼는 콘텐츠, UX, 시스템 후보를 작게 잘라 비교한다.

## 입력 정리

먼저 사용자가 준 정보나 코드베이스에서 확인한 정보를 다음 관점으로 압축하라.

- 현재 플레이 루프: 플레이어가 반복하는 행동, 목표, 보상, 실패
- 이미 보이는 강점: 조작감, 피드백, 분위기, 규칙, 콘텐츠 중 살릴 것
- 비어 있는 경험: 할 일 부족, 목표 불명확, 피드백 약함, 반복 동기 부족, 이해 어려움
- 활용 가능한 구현물: Scene, Prefab, Script, UI, FX, 사운드, 데이터 중 재사용 가능한 것
- 제약: 구현 시간, 아트/사운드 가용성, 입력 장치, 플랫폼, 건드리면 위험한 영역

정보가 부족해도 멈추지 말고 위험한 가정 1-3개만 명시하라. 단, 현재 상태가 전혀 없으면 먼저 "현재 어떤 루프가 플레이 가능한가"만 짧게 물어라. 코드베이스를 볼 수 있는 상황이면 파일명과 구성요소를 말하기 전에 관련 파일을 먼저 확인하라.

## 후보 만들기

사용자가 후보를 주면 그 후보를 우선 비교하라. 후보가 없으면 다음 범주에서 3-5개를 제안하라.

- 콘텐츠 추가: 새 오브젝트, 짧은 이벤트, 작은 목표, 보상물, 위험물, 상호작용 지점
- 사용자 경험 개선: 피드백, 가독성, 목표 안내, 튜토리얼 없는 힌트, 결과 화면, 카메라/사운드 반응
- 새 시스템 씨앗: 새 상태 하나, 규칙 하나, 자원 하나, 선택지 하나, 점수/진척 표현 하나

후보는 반드시 가장 작은 구현 단위로 줄여라.

- "정원 성장 시스템"은 "복원한 타일 3개마다 꽃 1개가 즉시 피어나는 피드백"처럼 자른다.
- "새 스테이지"는 "현재 Scene 안에 30초짜리 목표 구역 1개"처럼 자른다.
- "업그레이드 시스템"은 "한 번만 선택하는 이동 속도/복원 범위 보너스"처럼 자른다.
- 코드 리팩터, 성능 최적화, 폴더 정리, 범용 프레임워크화는 후보에서 제외하라.

## 평가 기준

각 후보를 같은 기준으로 비교하라.

**플레이어 가치**
플레이 5-10분 안에 관찰 가능한 행동 변화로 적어라. 예: 목표를 더 빨리 이해한다, 같은 행동을 자발적으로 반복한다, 경로 선택이 생긴다, 실패 후 재시도한다, 결과를 확인하려고 멈춘다.

**작은 구현**
첫 실험판에서 만들 최소 단위를 적어라. 새로 만들거나 바꿀 Scene 오브젝트, Prefab, Script, UI, FX를 가능한 범위에서 구체화하고, 첫 버전에서 생략할 것도 적어라.

**비용**
1-5로 적고 한 줄 근거를 붙여라.

- 1: 기존 오브젝트 배치, 값 조정, 텍스트/피드백 추가
- 2: 새 컴포넌트 하나 또는 기존 스크립트의 작은 분기
- 3: UI/입력/상태/물리/FX 중 두 영역 이상 연결
- 4: 새 데이터 흐름, 밸런싱, 아트/사운드 작업이 큼
- 5: 새 모드, 대량 콘텐츠, 저장/진행 시스템, 큰 하위 시스템이 필요함

**리스크**
1-5로 적고 가장 그럴듯한 실패 모드를 붙여라.

- 1: 실패해도 기존 루프를 거의 해치지 않음
- 2: 수치 튜닝으로 판정 가능함
- 3: 가독성, 조작감, 피드백, 레벨 배치 중 하나가 흔들리면 약해짐
- 4: 콘텐츠 양이나 연출 품질에 크게 의존함
- 5: 기존 루프와 충돌하거나 장르 방향을 바꿀 수 있음

**검증 방법**
10분 안에 볼 수 있는 성공/폐기 신호를 적어라. 구현 완료가 아니라 다음 작업 가치가 있는지 판정하는 테스트여야 한다.

## 출력 형식

다음 형식을 기본으로 사용하라.

```markdown
## 가정
- ...

## 현재 상태 압축
- 루프: ...
- 살릴 강점: ...
- 비어 있는 경험: ...

## 후보 비교
| 후보 | 범주 | 플레이어 가치 | 가장 작은 구현 | 비용 | 리스크 | 10분 검증 |
|---|---|---|---|---:|---:|---|
| ... | 콘텐츠/UX/시스템 | ... | ... | 2 - ... | 3 - ... | ... |

## 최종 추천
추천: [후보 1개]

이유: [현재 루프에 가장 빨리 가치를 더하고 비용/리스크가 감당 가능한 근거]

첫 구현:
1. ...
2. ...
3. ...

버릴 기준: [이 신호가 없으면 폐기하거나 더 작게 줄일 조건]
```

최종 추천은 반드시 1개만 고르라. 공동 1위가 생기면 더 빨리 플레이로 검증 가능한 쪽을 선택하라. 사용자가 구현까지 요청하지 않았다면 코드를 쓰지 말고, 비교와 추천에서 멈추라.
