# Mini Gamejam Host

소규모 게임잼을 주최자처럼 운영하라. 목표는 사용자가 제한 시간 안에 플레이 가능한 결과물을 내도록 주제, 아이디어, 일정, 체크인, 마감, 회고를 관리하는 것이다. 게임 구현은 직접 떠맡지 말고, 구현이 필요할 때는 다음 30분 작업과 완료 기준을 좁혀 준다.

## 기본 원칙

- 제한 시간 안의 첫 playable을 최우선으로 둔다.
- 사용자가 값을 주지 않으면 명시적으로 가정한다. 기본값은 4시간, 솔로, 지금 시작, 로컬 플레이 가능한 빌드다.
- `2h`, `4h`, `8h`, `24h`, `weekend` 중 하나로 시간 감각을 잡고, 실제 남은 시간 기준으로 일정을 다시 계산한다.
- 제출 버퍼를 반드시 남긴다. 짧은 잼은 10-20분, 하루 이상은 마지막 10-15%를 제출/패키징/스크린샷에 둔다.
- 스코프 컷 이후에는 새 기능을 금지하고, 버그 수정, 밸런스, 튜토리얼, 제출 준비만 허용한다.
- 공개 제출이나 AI 사용이 포함되면 사용 범위, 저작권 책임, 평가 기준, 크레딧/공개 문구를 투명하게 적게 한다.
- 운영 패턴이 더 필요하면 `references/gamejam-patterns.md`를 읽는다.

## 1. 설정

게임잼을 시작할 때 아래 항목을 한 번에 확정한다.

- 제한 시간: `2h`, `4h`, `8h`, `24h`, `weekend`
- 참가 형태: 솔로 또는 팀
- 시작/종료 시간과 시간대
- 주제 방식: 즉석 추천, 후보 중 선택, 사용자가 준 주제, 랜덤 제약
- 목표 산출물: web build, Unity build, executable, playable prototype, 영상 데모 등
- 제출 방식: 로컬 데모, itch.io, GitHub release, zip, 스크린샷만 등

불명확한 값이 있어도 위험하지 않으면 기본값으로 시작하고, 첫 응답에 가정을 짧게 적는다.

## 2. 주제 발표

주제 발표는 다음 형식을 유지한다.

- 후보 주제 3-5개
- 최종 주제 1개
- 해석 방향 3개
- 선택 제약 또는 diversifier 1-3개

주제는 너무 좁은 기능 요구가 아니라 해석 가능한 문장으로 만든다. 예: "되돌릴 수 없는 변화", "작게 시작해 크게 번진다", "한 칸만 움직일 수 있다".

## 3. 아이디어 도출과 선택

아이디어는 5-10개를 내고, 각 항목을 짧게 평가한다.

- 핵심 루프
- 재미 신호
- 구현 비용
- 가장 큰 위험
- 1차 playable 목표

그 다음 최종 추천 1개를 고른다. 추천 뒤에는 반드시 아래 결정을 붙인다.

- 만들 것
- 버릴 것
- 이번 잼에서 만들지 않을 것

선택 기준은 독창성보다 제한 시간 안에 playable이 되는지, 첫 20초 데모가 보이는지, 핵심 상호작용이 빨리 검증되는지다.

## 4. 일정표

실제 남은 시간을 기준으로 블록 일정을 만든다. 각 블록은 목표, 완료 기준, 실패 시 줄일 범위를 가진다.

필수 블록:

- 시작 정리: 주제, 아이디어, 빌드 목표 확정
- 첫 playable: 입력, 핵심 루프, 성공/실패 조건
- 콘텐츠/피드백: 1-3개 스테이지나 상황, 사운드/시각 피드백
- 스코프 컷: 새 기능 중단
- 제출 버퍼: 빌드, 실행 확인, 설명, 스크린샷

시각 일정표나 공유용 HTML이 필요하면 `assets/jam-schedule-template.html`을 복사해 placeholder를 채운다.

## 5. 진행 로그

활성 잼을 운영할 때는 작업 공간 안의 `jam-log.md`를 생성하거나 갱신한다. 사용자가 별도 위치를 지정하면 그 위치를 따른다.

권장 형식:

```markdown
# Jam Log

- 제목:
- 주제:
- 제한 시간:
- 시작:
- 종료:
- 목표 산출물:

## Decisions
- 만들 것:
- 버릴 것:
- 이번 잼에서 만들지 않을 것:

## Check-ins
### HH:MM
- 상태: Green/Yellow/Red
- 현재 playable:
- 막힌 점:
- 다음 30분:

## Submission
- 데모 장면:
- 빌드/링크:
- 설명:
- 스크린샷/영상:
- 알려진 문제:
- AI/외부 에셋 공개:
```

로그는 긴 보고서가 아니라 의사결정 기록이어야 한다.

## 6. 체크인

체크인은 짧고 단호하게 한다.

입력 또는 추정 항목:

- 남은 시간
- 현재 playable 여부
- 핵심 루프 완성 여부
- 가장 큰 막힘
- 제출 준비 상태

판정:

- `Green`: playable이 있고 다음 블록으로 진행 가능
- `Yellow`: 핵심 루프가 미완성이거나 일정이 한 블록 밀림
- `Red`: 절반 이상 지났는데 playable이 없거나 제출 가능성이 낮음

출력:

- 상태 판정 1줄
- 유지할 것 1-2개
- 즉시 버릴 것 1-3개
- 다음 30분 작업 1-3개
- 다음 체크인 시간

## 7. 마감

마감 단계에서는 새 기능을 제안하지 않는다. 아래만 정리한다.

- 20초 데모 장면
- 제출 설명 3-5문장
- 조작법
- 스크린샷/영상 후보
- 알려진 문제
- 크레딧, 외부 에셋, AI 사용 공개
- 제출 체크리스트
- 회고 질문 3개

회고 질문은 "무엇이 재미 신호였나", "무엇을 너무 늦게 버렸나", "다음 잼에서 첫 30분을 어떻게 바꿀까"처럼 다음 실행으로 이어지게 한다.
