# Gamejam 운영 패턴

이 문서는 특정 잼의 규칙을 복제하지 않고, 여러 공개 게임잼에서 반복되는 운영 패턴만 추상화한다.

## 공통 패턴

- 주제: 시작 시점에 공개하고, 참가자가 여러 방식으로 해석할 수 있게 둔다.
- 제약: 주제와 별도로 선택형 diversifier나 제한 조건을 제공하면 아이디어가 빨리 좁혀진다.
- 시간 압축: 첫 playable, 스코프 컷, 제출 버퍼를 일정의 고정 지점으로 둔다.
- 제출: 실행 가능한 빌드, web embed, zip, 스크린샷, 조작법, 알려진 문제를 명확히 요구한다.
- 평가: 주제 적합성, 재미, 창의성, 완성도, 프레젠테이션처럼 적은 수의 기준을 미리 정한다.
- 피드백: 경쟁보다 플레이/댓글/스트림/회고를 강조하면 짧은 잼의 부담이 줄어든다.
- AI 사용: 허용 여부보다 사용 범위, 저작권 책임, 공개 문구, 평가 영향 여부를 분명히 적는다.
- 사전 준비: 엔진, 템플릿, 계정, 빌드 파이프라인, 에셋 라이선스를 시작 전에 확인한다.

## 시간대별 운영 감각

- 1-3시간: 한 가지 상호작용과 한 화면만 목표로 둔다. 제출 버퍼를 최소 10분 둔다.
- 4-8시간: 첫 25% 안에 playable을 만든다. 중간 지점에서 콘텐츠 수를 잘라 낸다.
- 24시간: 수면/식사와 빌드 검증 시간을 일정에 넣는다. 마지막 2-3시간은 제출 안정화로 둔다.
- 주말: 금요일은 주제/팀/프로토타입, 토요일은 playable과 콘텐츠, 일요일은 polish와 제출로 나눈다.

## 진행자가 반복해서 확인할 질문

- 지금 실행 가능한 playable이 있는가?
- 핵심 루프가 20초 안에 보이는가?
- 지금 추가하려는 기능이 제출 가능성을 높이는가?
- 남은 시간 안에 버려야 할 콘텐츠는 무엇인가?
- 제출 페이지를 처음 보는 사람이 조작법과 목표를 이해할 수 있는가?
