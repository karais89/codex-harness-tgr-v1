# Microgame Mechanic Audition

## 목표

작은 게임 프로토타입에 다음으로 넣을 "하나의 작은 playable mechanic"을 고르는 데 집중하라. 콘텐츠 확장, 장기 로드맵, 큰 시스템 설계가 아니라 짧게 만들고 플레이로 판별할 수 있는 후보를 비교하라.

## 입력 정리

먼저 사용자가 준 정보를 다음 관점으로 압축하라.

- 현재 핵심 루프: 플레이어가 반복해서 하는 행동과 보상
- 기존 조작/카메라/장르: 새 메커닉이 끼어들 위치
- 이미 있는 Unity 구성: Scene, Prefab, Script, Collider, UI, Animator, Input 중 활용 가능한 것
- 원하는 재미 방향: 긴장, 속도감, 발견, 숙련, 리스크/보상, 촉감 등
- 제약: 구현 시간, 아트/사운드 가용성, 모바일/키보드/패드 입력, 코드 변경 범위

정보가 부족해도 멈추지 말라. 위험한 가정은 1-3개만 명시하고, 그 가정 아래에서 비교하라. 단, 기존 코드베이스를 볼 수 있는 상황이면 파일명과 Unity 구성요소를 말하기 전에 먼저 관련 파일을 확인하라.

## 후보 만들기

사용자가 후보를 주면 그 후보를 우선 비교하라. 후보가 없으면 현재 루프에 붙일 수 있는 후보 3-5개를 직접 제안하라.

후보는 다음 조건을 만족해야 한다.

- 새 입력 하나, 새 규칙 하나, 새 피드백 하나로 설명될 만큼 작아야 한다.
- 기존 루프의 한 순간을 더 재미있게 만들어야 한다.
- 새 모드, 긴 진행 시스템, 대량 콘텐츠, 복잡한 AI, 큰 UI 화면을 요구하면 후보에서 제외하거나 더 작은 형태로 줄여라.
- "아이템 추가", "스테이지 추가", "더 많은 적"처럼 콘텐츠 양만 늘리는 제안은 마이크로 메커닉으로 취급하지 말라.

## 평가 기준

각 후보를 같은 기준으로 비교하라.

**재미 신호**
플레이 5-10분 안에 관찰할 수 있는 신호로 적어라. 예: 플레이어가 자발적으로 반복한다, 실패 직후 다시 시도한다, 경로 선택이 달라진다, 타이밍을 조절한다, 리스크를 감수한다, 피드백을 기다린다.

**구현 비용**
1-5로 적고 한 줄 근거를 붙여라.

- 1: 값 조정, 기존 컴포넌트 연결, 아주 작은 조건문
- 2: 새 MonoBehaviour 하나 또는 기존 스크립트의 작은 분기
- 3: 물리/입력/UI/상태 중 두 영역 이상을 연결
- 4: 애니메이션, 이펙트, 레벨 데이터, 저장, 튜닝 도구 등 보조 작업이 큼
- 5: 새 하위 시스템이나 대량 콘텐츠가 필요함

**망할 가능성**
1-5로 적고 가장 그럴듯한 실패 모드를 붙여라.

- 1: 피드백이 즉시 보이고, 실패해도 기존 루프를 거의 해치지 않음
- 2: 수치 튜닝이 필요하지만 관찰이 쉬움
- 3: 조작감, 판정, 타이밍, 가독성 중 하나가 흔들리면 재미가 사라짐
- 4: 아트/사운드/레벨 디자인/콘텐츠 양에 크게 의존함
- 5: 장르 방향을 바꾸거나 기존 루프와 충돌할 가능성이 큼

**가장 작은 Unity 구현 방식**
실험판 기준으로 적어라. 파일명이나 컴포넌트를 확정할 수 있으면 구체적으로 쓰고, 확정할 수 없으면 일반화해서 쓰라.

- 새로 만들거나 바꿀 Script/Prefab/Scene 오브젝트
- 필요한 Collider/Rigidbody/Trigger/Input/UI 피드백
- 첫 구현에서 생략할 것
- 10분 플레이 검증 방법

## 출력 형식

다음 형식을 기본으로 사용하라. 장황한 설명보다 비교 가능성이 우선이다.

```markdown
## 가정
- ...

## 후보 비교
| 후보 | 재미 신호 | 구현 비용 | 망할 가능성 | 가장 작은 Unity 구현 | 10분 검증 |
|---|---|---:|---:|---|---|
| ... | ... | 2 - ... | 3 - ... | ... | ... |

## 최종 추천
추천: [후보 1개]

이유: [재미 신호 대비 비용/리스크가 가장 낫다는 근거]

첫 구현: [가장 작은 Unity 작업 순서 3-5개]

버릴 기준: [이 신호가 없으면 폐기하거나 축소할 조건]
```

최종 추천은 반드시 1개만 고르라. 공동 1위가 생기면 더 작은 구현으로 검증 가능한 쪽을 선택하라.

## 판단 규칙

- 재미 신호가 관찰 불가능하면 낮게 평가하라.
- 구현 비용이 낮아도 실패 모드가 루프 전체를 망치면 추천하지 말라.
- "재미있을 것 같다"보다 "어떤 행동이 늘어나면 재미 신호로 볼 것인가"를 우선하라.
- Unity 구현은 완성 기능이 아니라 가장 작은 실험판으로 제안하라.
- 사용자가 구현까지 요청하지 않았다면 코드를 쓰지 말고, 비교와 추천에서 멈추라.
