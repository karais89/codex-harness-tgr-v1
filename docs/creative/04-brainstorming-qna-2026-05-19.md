# 브레인스토밍 Q&A 원문 기록

상태: 기록 완료

## 목적

이 문서는 2026-05-19 세션에서 Tiny Garden Restore 기획 브레인스토밍 중 Codex가 던진 질문과 사용자의 직후 답변을 원문 그대로 보존하기 위한 별도 기록이다.

이 문서는 확정 구현 사양이 아니다. Player-facing design 기준으로 쓰려면 `docs/design` 문서로 별도 정리하고 사용자 승인을 받아야 한다.

## 기록 범위

- Codex가 사용자에게 던진 브레인스토밍 질문
- 질문에 포함된 Codex 추천 답안, 예시, 이유
- 사용자의 직후 답변

## Q&A

### Q1. 중심 재미

Codex 질문 원문:

````text
**이 게임의 중심 재미를 어디에 두고 싶나요?**

내 추천 답안: **“복구 순서와 상태 변화가 보이는 작은 퍼즐형 정원 복구”**
이유는 TGR v1의 목표인 기획, 상태, 데이터, 해금, 검증 흐름을 가장 작게 테스트할 수 있기 때문입니다.
````

사용자 답변 원문:

```text
복구 순서와 상태 변화가 보이는 작은 퍼즐형 정원 복구
```

### Q2. 퍼즐 긴장감

Codex 질문 원문:

````text
**질문:** 이 퍼즐은 어떤 긴장감이 좋을까요?

내 추천 답안: **제한된 행동 횟수 안에서 복구 순서를 고르는 방식**입니다.
이유는 자원 경제를 크게 만들지 않아도 되고, 첫 playable에서 “순서 선택 → 상태 변화 → 결과 확인”이 가장 선명하게 검증됩니다.
````

사용자 답변 원문:

```text
제한된 행동 횟수 안에서 복구 순서를 고르는 방식
```

### Q3. 복구 순서가 중요해지는 원인

Codex 질문 원문:

````text
**질문:** 복구 순서는 무엇 때문에 중요해지는 게 좋을까요?

내 추천 답안: **접근 경로 해금형**입니다.

예:

```text
잡초 더미 제거
-> 꽃밭 접근 가능
-> 꽃밭 복구
-> 분수 주변 길 열림
-> 분수 복구
```

이 방식이 첫 playable에 가장 좋습니다. 이유는 플레이어가 “왜 지금 이걸 못 고르는지”를 화면에서 이해하기 쉽고, 데이터/상태/해금 검증도 작게 만들 수 있기 때문입니다.
````

사용자 답변 원문:

```text
접근 경로 해금형
```

### Q4. 공간 표현

Codex 질문 원문:

````text
**질문:** 정원의 공간 표현은 어떤 방식이 좋을까요?

내 추천 답안: **작은 노드 맵 방식**입니다.

예:

```text
[입구] - [잡초] - [꽃밭]
             |
          [깨진 길] - [분수]
```

이유는 격자 이동보다 구현과 검증이 작고, 접근 경로 해금이 눈에 잘 보이며, 데이터로 `prerequisiteObjectIds` 같은 조건을 다루기 좋기 때문입니다.
````

사용자 답변 원문:

```text
작은 노드 맵 방식
```

### Q5. 첫 playable 목표

Codex 질문 원문:

````text
**질문:** 첫 playable에서 플레이어의 목표는 무엇이 좋을까요?

내 추천 답안: **제한 행동 안에 핵심 정원 오브젝트 1개를 복구하기**입니다.

예:

```text
최종 목표: 작은 분수 복구
부가 목표: 꽃밭까지 복구하면 더 좋은 결과
```

이유는 “모든 걸 복구해야 성공”으로 잡으면 첫 버전의 퍼즐 밸런스가 바로 빡빡해지고, 반대로 핵심 목표 1개를 두면 성공 조건과 선택 여지를 작게 둘 수 있습니다.
````

사용자 답변 원문:

```text
제한 행동 안에 핵심 정원 오브젝트 1개를 복구
```

### Q6. 핵심 정원 오브젝트

````text

````

### Q7. 노드 복구 단순도

Codex 질문 원문:

````text
**각 노드 복구는 얼마나 단순하게 갈까요?**

내 추천 답안: **첫 playable에서는 모든 노드가 `복구하기` 1회 행동만 사용한다**입니다.
이유는 자원 종류나 작업 타입을 넣으면 퍼즐의 초점이 흐려지고, 지금 검증하려는 “접근 경로 해금 + 상태 변화 + 제한 행동”만으로도 충분히 작은 퍼즐이 됩니다.
````

```text
추천 답안으로
```

### Q8. 행동 횟수

Codex 질문 원문:

````text
**행동 횟수는 어느 정도가 좋을까요?**

````


### Q9. 첫 노드 맵 크기

Codex 질문 원문:

````text
**질문:** 첫 노드 맵의 크기는 어떻게 잡을까요?

내 추천 답안: **6개 노드, 그중 4개가 분수 성공 경로**입니다.

예:

```text
[입구]
  |
[잡초 더미]
 /        \
[꽃밭]   [낡은 벤치]
  |
[깨진 길]
  |
[작은 분수]
```

성공 경로:

```text
잡초 더미 -> 꽃밭 -> 깨진 길 -> 작은 분수
```

낡은 벤치는 우회/함정/부분 성과 노드로 둡니다.
이유는 4회 행동 제한과 잘 맞고, “눈에 보이는 선택지 하나가 있지만 목표에는 직접 도움 되지 않는다”는 퍼즐 감각을 아주 작게 만들 수 있습니다.
````

사용자 답변 원문:

```text
추천댈고
```

### Q10. 낡은 벤치 역할

Codex 질문 원문:

````text
**낡은 벤치는 어떤 역할이 좋을까요?**

내 추천 답안: **실패를 즉시 만들지는 않지만, 분수 성공 경로에서 벗어나는 “부분 성과 노드”**입니다.
즉, 벤치를 복구하면 화면은 좋아지지만 4회 제한 안에 분수까지는 못 가게 됩니다. 이러면 플레이어가 “상태 변화는 좋지만 목표에는 맞지 않았다”를 바로 배울 수 있습니다.
````

### Q11. 플레이어에게 보이는 노드 상태 단계

Codex 질문 원문:

````text
**노드 상태는 플레이어에게 몇 단계로 보이는 게 좋을까요?**

내 추천 답안: **3단계만 보이게 하기**입니다.

```text
잠김
복구 가능

```

이유는 첫 playable에서 `Cleaned`, `Restored`, `Available` 같은 세부 상태를 많이 보여주면 퍼즐보다 상태 용어가 먼저 복잡해집니다. 내부 데이터는 나중에 늘릴 수 있지만, 플레이어가 보는 상태는 처음엔 3단계가 좋습니다.
````

사용자 답변 원문:

```text
추천대로
```

### Q16. 노드 이름 방향

Codex 질문 원문:

````text
**첫 playable에서 노드 이름은 현실적인 정원 오브젝트로 갈까요, 아니면 조금 동화적인 이름을 붙일까요?**

내 추천 답안: **현실적인 이름을 기본으로 하고, 분위기만 부드럽게 가져가기**입니다.

꽃밭

깨진 길
작은 분수
```

이유는 첫 playable에서는 퍼즐 규칙을 바로 이해하는 게 중요해서, 이름이 너무 시적이면 기능을 알아보기 어려워질 수 있습니다.
````

사용자 답변 원문:

```text
추천대로
```

### Q17. 정원 분위기

Codex 질문 원문:

````text
**정원의 분위기는 어느 쪽이 좋을까요?**

내 추천 답안: **버려졌지만 무섭지 않은, 조용하고 따뜻한 정원**입니다.

예:

```text
- 회색빛으로 방치된 정원에서 시작
- 복구할수록 초록색과 밝은 색이 늘어남
- 어둡거나 폐허 느낌보다는 “다시 돌보면 살아날 공간”에 가까움
```

이유는 작은 퍼즐이어도 “복구했다”는 감정이 잘 살아나고, 향후 art direction으로 넘기기도 좋기 때문입니다.
````

사용자 답변 원문:

```text
추천대로
```
