# Creative 문서 규칙

## 역할

`docs/creative`는 gameplay design 이전의 가벼운 컨셉 탐색 공간이다.

이 폴더에서는 게임이 어떤 느낌인지, 어떤 재미를 상상하는지, 어떤 정서와 소재를 다루고 싶은지, 어떤 방향을 피하고 싶은지를 탐색한다. 모호함과 여러 가능성을 허용하며, 빠르게 첫 playable 사양으로 압축하는 것이 목표가 아니다.

Creative 산출물은 이후 `docs/design` 문서의 재료가 될 수 있지만, 그 자체로 gameplay ExecPlan이나 Unity 구현 기준이 되지 않는다.

## 먼저 읽기

Codex는 다음 작업을 시작하기 전에 이 문서를 읽는다.

- 게임 컨셉 브레인스토밍
- 정서, 감각, 소재, 재미 후보 탐색
- 좋아하는 방향과 피하고 싶은 방향 기록
- concept 후보 정리
- concept brief 작성 또는 갱신
- creative 결과를 design 기준으로 넘길지 판단하는 작업

## Creative에서 다루는 것

Creative 단계에서는 다음을 다룬다.

- 플레이어에게 남기고 싶은 감정과 인상
- 게임이 떠올리게 하는 장면, 소재, 상황
- 재미의 넓은 후보
- 게임을 만들 때 붙잡고 싶은 생각
- 피하고 싶은 톤, 장르 관습, 감각
- 아직 열어 두고 싶은 가능성
- art 또는 design으로 넘길 만한 질문

## Creative에서 확정하지 않는 것

Creative 단계에서는 다음을 확정하지 않는다.

- 행동 횟수, 자원량, 점수 같은 수치
- 성공 조건과 실패 조건
- 조작 방식과 UI 버튼
- 플레이어에게 보이는 상태명
- 맵 구조, 노드 수, 레벨 구조
- 힌트, 결과 메시지, 보상 화면 같은 구체 피드백
- Unity 구현 방식, 데이터 구조, 에셋 사용 여부

이 항목들은 player-facing gameplay 기준이므로 필요하면 `docs/design` 절차로 넘긴다.

## 문서 흐름

Creative 작업은 기본적으로 다음 단계별 문서에 기록한다.

1. `01-brainstorming.md`: 컨셉 감각, 정서, 소재, 재미 후보를 넓게 기록한다.
2. `02-concept-candidates.md`: 후보의 매력, 우려, 보류/폐기 이유를 기록한다.
3. `03-concept-brief.md`: 현재 채택한 컨셉 초안을 짧게 정리한다.

각 문서는 `상태: 작성 전`으로 시작할 수 있다. 이 상태의 문서는 아직 실제 기획 산출물이 아니라 placeholder다.

## 엄격 모드

Creative 작업 중 다음이 애매하면 Codex는 파일을 수정하지 않고 사용자에게 먼저 질문한다.

- 가벼운 컨셉 기록인지, player-facing design 기준인지
- 감정과 방향 탐색인지, 수치나 조작 같은 규칙 확정인지
- 새 후보를 추가할지, 기존 후보를 선택/보류/폐기할지
- concept brief를 갱신할지, 후속 design 문서 생성을 논의할지

질문 예:

```text
이 내용은 creative의 컨셉 감각 기록으로만 남길까요,
아니면 플레이어 규칙을 정하는 docs/design 논의로 넘길까요?
```

## Design으로 넘어가는 조건

Creative 문서의 내용이 플레이어 경험, 게임 규칙, 목표, 조작, 피드백, 화면 상태, 구현 기준을 바꾸면 `docs/design` 갱신 절차로 넘어가야 한다.

이때 Codex는 creative 문서를 그대로 확정 사양처럼 사용하지 않는다. 먼저 사용자와 공유된 이해를 검증하고, 명시 승인된 내용만 `docs/design` 문서에 반영한다.

## 금지 사항

- Creative 아이디어를 근거로 gameplay 구현을 바로 시작하지 않는다.
- Creative 브레인스토밍 중 행동 횟수, 상태명, 조작 방식, 성공/실패 조건을 확정하지 않는다.
- `03-concept-brief.md`를 승인된 design 문서처럼 사용하지 않는다.
- 폐기 또는 보류된 아이디어를 조용히 구현 범위에 포함하지 않는다.
- 사용자 승인 없이 creative 산출물을 `상태: 승인됨`인 design 기준으로 바꾸지 않는다.
