# project-goal-v2.md

## 프로젝트 이름

codex-harness-tgr-v1

## 게임 이름

Tiny Garden Restore

## v2 한 줄 정의

Tiny Garden Restore v2는 v1 playable loop를 유지한 상태에서 실제 Unity Scene 에셋을 교체하고, 최소 UX 피드백을 보강하며, Codex 반복 실수를 줄이기 위한 경량 AI 실수 장부를 함께 운영하는 후속 마일스톤입니다.

## v2 목적

v2의 비중은 다음으로 둡니다.

```text
게임 개선 60%
하네스 / AI 실수 관리 40%
```

게임 개선은 이미지 교체와 UX 피드백 보강을 뜻합니다. 하네스 / AI 실수 관리는 `docs/ai-mistakes.md`, `tiny-garden-playmode-check` skill, Unity MCP preflight hook 판단, Game view 검증 기준을 뜻합니다.

v2는 큰 게임 확장이 아닙니다. v1에서 완성한 작은 playable loop는 유지하고, 플레이어가 정원 상태를 더 잘 읽게 만드는 시각/피드백 개선과 Codex가 같은 실수를 반복하지 않도록 하는 경량 운영 장치를 추가합니다.

## v1 결과 요약

v1은 `docs/project-goal.md` 기준으로 `M0`부터 `M7`까지 종료 선언되었습니다.

- `M5`에서 작은 playable loop가 완료되었습니다.
- Player는 작은 정원 안에서 이동할 수 있습니다.
- `잡초`, `시든 꽃`, `더러운 타일` 세 복구 대상이 시작 시점부터 Scene에 존재합니다.
- Player는 각 대상 근처에서 `E` 1회 입력으로 복구 상호작용을 수행합니다.
- 세 대상이 모두 복구되면 완료 상태가 표시됩니다.
- `M6`에서 경량 하네스는 v1 목적에 충분하다고 평가되었습니다.
- `M7`에서 `tiny-garden-playmode-check` skill은 유지 후보로 남았고, 보고서 기준 Hooks는 보류로 판정되었습니다.

v2 시작 전에는 현재 `.codex/config.toml`, `.codex/hooks/unity_mcp_preflight.py`, `.codex/skills/tiny-garden-playmode-check/SKILL.md` 상태를 다시 확인합니다. 특히 M7 보고서의 "Hooks 보류"와 현재 hook 존재 여부가 충돌하면, 유지, 수정, 제거, 문서화 중 하나로 판단합니다.

## v2 확정 범위

### 1. AI 실수 장부 생성

v2 시작 시점에 `docs/ai-mistakes.md`를 바로 생성합니다.

이 문서는 memory-bank가 아니라 경량 재발 방지 장부입니다. 기록 단위는 크게 만들지 않고 다음 항목만 둡니다.

```text
- 실수
- 영향
- 재발 방지 규칙
- 관련 근거
```

첫 기록 대상은 이번 v2 문서 작성 과정에서 확인된 "사용자 의도보다 v2 범위를 좁게 해석해 이미지 교체와 AI 실수 장부를 누락한 문제"입니다.

### 2. 이미지 교체 기준 확정

이미지 교체는 v2의 명시적 범위입니다. 단, 바로 Unity에 넣지 않고 다음 순서를 지킵니다.

```text
기존 v1 아트 기준 확인
-> v2 이미지 교체 목적 정의
-> 복구 대상별 이미지 프롬프트 작성
-> 이미지 생성
-> Unity 사용 가능성 확인
-> Scene 교체
```

필수 교체 대상은 다음입니다.

- `잡초`
- `시든 꽃`
- `더러운 타일`

선택 교체 대상은 다음입니다.

- Player

Player는 이미지 생성 결과와 작업량을 보고 교체 여부를 결정합니다.

### 3. 실제 Unity Scene 에셋 교체

v2는 이미지 후보 생성이나 문서화에서 끝나지 않습니다. 선택한 이미지를 Unity 프로젝트에 넣고 실제 Scene에서 기존 복구 대상 표현을 교체합니다.

완료 판정은 문서나 에셋 파일 존재만으로 하지 않습니다. Unity Scene에서 읽히는지 확인해야 합니다.

### 4. 최소 UX 피드백 보강

UX 피드백 보강은 이미지 교체 뒤에 진행합니다.

대상은 다음입니다.

- 정원 회복도 `0/3`, `1/3`, `2/3`, `3/3` 표시
- 대상 근처에서 `E: Restore` 표시
- 복구 후 짧은 시각 피드백

UX 피드백은 새 시스템이 아니라 읽힘 보강입니다. 이미지 교체보다 앞서서 규칙을 키우지 않습니다.

### 5. Game view / Play Mode 검증 보강

v2에서는 로직 통과와 화면 판독을 분리합니다.

- Play Mode: 시작, 이동, 세 대상 복구, 완료 상태, Console error를 확인합니다.
- Game view 또는 screenshot: 교체된 이미지, 진행도, `E: Restore`, 완료 표시가 실제로 읽히는지 확인합니다.
- MCP나 smoke runner로 확인하지 못한 실제 조작감은 `미확인`으로 분리합니다.

## v2에서 하지 않을 것

v2는 다음을 기본 범위로 삼지 않습니다.

- 인벤토리
- 물뿌리개, 삽, 빗자루 같은 도구 선택
- 대상별 도구 매칭
- 복구 시간 게이지
- 스태미나
- 제한 시간
- 점수, 별점, 콤보
- 경제, 성장, 상점
- 하루/밤 루프
- 여러 정원, 여러 스테이지
- 세이브/로드
- 랜덤 배치
- 복구 순서 퍼즐
- 큰 하네스 체계
- memory-bank 또는 복잡한 workflow 문서
- 승인 없는 게임 규칙 변환

위 항목은 별도 승인 없이는 v2 기본 범위에 넣지 않습니다.

## v2 마일스톤

v2 체크리스트는 기존 v1 `docs/milestones/M0-checklist.md`부터 `M7-checklist.md`와 섞지 않고 다음 폴더에 둡니다.

```text
docs/milestones/v2/
```

### V2-M0. v2 범위 재확정과 현재 상태 확인

목표는 사용자와 확정한 v2 범위가 현재 문서에 반영되어 있는지 확인하고, v1 종료 결과와 현재 hook/skill 상태를 다시 고정하는 것입니다.

완료 기준: v2의 3축인 이미지 교체, AI 실수 장부, UX/검증 보강이 모두 명시되어 있고, 현재 작업 트리와 `.codex` 상태를 설명할 수 있어야 합니다.

### V2-M1. AI 실수 장부 생성

목표는 `docs/ai-mistakes.md`를 생성하고 첫 기록을 남기는 것입니다.

완료 기준: `docs/ai-mistakes.md`가 존재하며, 기록 형식과 첫 실수 사례가 재발 방지 규칙까지 포함해 작성되어 있어야 합니다.

### V2-M2. 이미지 교체 기준 확정

목표는 기존 v1 아트 기준을 확인하고, v2 이미지 교체 목적과 대상별 프롬프트 기준을 문서로 정리하는 것입니다.

완료 기준: 필수 대상 3개와 선택 대상 Player의 교체 여부 기준, 이미지 생성 프롬프트, Unity 사용 가능성 판단 기준이 정리되어 있어야 합니다.

### V2-M3. 이미지 생성과 선정

목표는 V2-M2 기준에 따라 실제 이미지 후보를 생성하고 Unity에 넣을 후보를 고르는 것입니다.

완료 기준: 필수 복구 대상 3개의 이미지 후보가 생성되어 있고, 선정/보류 이유와 Unity 사용 가능성 판단이 기록되어 있어야 합니다.

### V2-M4. Unity Scene 에셋 교체

목표는 선정한 이미지를 Unity 프로젝트에 넣고 Scene의 복구 대상 표현을 실제로 교체하는 것입니다.

완료 기준: Unity Scene에서 `잡초`, `시든 꽃`, `더러운 타일`의 새 이미지가 표시되고, 기존 playable loop가 깨지지 않아야 합니다.

### V2-M5. 최소 UX 피드백 보강

목표는 이미지 교체 뒤 정원 상태와 상호작용 가능 여부가 더 잘 읽히도록 작은 UX 피드백을 추가하는 것입니다.

완료 기준: Play Mode에서 `0/3`부터 `3/3`까지 진행도, 필요한 경우 `E: Restore`, 복구 후 짧은 시각 피드백이 확인되고 Console error가 없어야 합니다.

### V2-M6. Game view / Play Mode 검증 보강

목표는 v2 결과가 로직으로도 통과하고 화면으로도 읽히는지 확인하는 것입니다.

완료 기준: Play Mode 검증 결과와 Game view 또는 screenshot 기반 시각 검수 결과가 분리되어 기록되어야 합니다.

### V2-M7. v2 결과 평가

목표는 v2 보강이 실제로 v1보다 나아졌는지 판단하는 것입니다.

완료 기준: 이미지 교체, UX 피드백, AI 실수 장부, skill/hook 유지 가치에 대해 유지할 것, 되돌릴 것, 다음 프로젝트로 넘길 것이 구분되어 있어야 합니다.

## 검증 기준

v2 작업은 다음 기준을 만족해야 완료로 봅니다.

- `git status --short --untracked-files=all`로 작업 범위를 확인합니다.
- 기존 v1 완료 기준을 깨지 않습니다.
- 이미지 교체는 실제 Unity Scene 반영까지 확인합니다.
- Play Mode에서 Player 이동, 세 대상 복구, 완료 상태를 확인합니다.
- Console error가 없습니다.
- Game view 또는 screenshot으로 교체 이미지와 UX 피드백이 읽히는지 확인합니다.
- `docs/ai-mistakes.md`는 실제 재작업이나 품질 저하로 이어진 구체적 실수만 기록합니다.
- `tiny-garden-playmode-check` skill을 사용할 경우, 미확인 항목을 통과처럼 쓰지 않습니다.
- Unity MCP preflight hook은 유지, 수정, 제거, 문서화 중 하나로 판단하고 이유를 남깁니다.

## 하네스 판단

v2의 기본 하네스는 v1보다 크게 만들지 않습니다.

- `AGENTS.md`는 유지합니다.
- Codex 기본 Plan mode는 유지합니다.
- Unity MCP는 유지합니다.
- ExecPlan은 복합 기능이나 큰 리팩터링에서만 참조합니다.
- `docs/ai-mistakes.md`는 경량 재발 방지 장부로만 둡니다.
- `tiny-garden-playmode-check` skill은 Unity 검증 루틴의 유지 후보로 둡니다.
- Unity MCP preflight hook은 현재 존재하더라도 v2 기본값으로 자동 확정하지 않습니다.
