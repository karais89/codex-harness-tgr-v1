# Design 문서 작성 규약

## 역할

`docs/design`은 승인된 플레이어 경험, 게임 방향, 게임 규칙 기준을 보관하는 폴더다.

이 폴더는 단계 계획 문서 폴더가 아니다. 단계 계획은 언제 무엇을 구현할지 다루고, design 문서는 플레이어가 무엇을 경험해야 하는지와 어떤 게임 규칙을 지켜야 하는지를 다룬다.

`docs/design/README.md`는 player-facing gameplay design 문서가 아니라 design 문서 작성 규약이다. 따라서 `docs/design/README.md`는 승인 상태 확인 대상이 아니라 존재 확인 대상이다.

`docs/design/*.md` 파일은 실제 design 산출물이다. `PLANS.md`가 ExecPlan 작성 규약이고 `exec-plans/*.md`가 실제 실행 계획인 것처럼, `docs/design/README.md`는 design 문서 작성 규약이고 `docs/design/*.md`는 실제 design 산출물이다.

Design 작업은 `docs/workflows/grill-me.md`의 `Grill-Me Discovery Protocol`을 따른다. Codex는 player-facing 기준을 추론해서 바로 쓰지 않고, 사용자의 암묵적 design 판단 기준과 제외 범위를 공유된 이해가 닫힐 때까지 질문한다.

## 문서 상태

실제 design 산출물은 문서 상단에 다음 상태 중 하나를 적는다.

- `상태: 작성 전`: 파일은 있을 수 있지만 아직 인터뷰와 공유된 이해 검증이 끝나지 않았다. 이 상태의 파일은 design 산출물이 아니며 gameplay ExecPlan이나 구현의 승인 기준으로 사용할 수 없다.
- `상태: 초안`: 인터뷰와 공유된 이해 검증을 거쳐 작성됐지만 아직 사용자 승인 전이다. gameplay ExecPlan이나 구현의 승인 기준으로 사용할 수 없다.
- `상태: 승인됨`: 사용자가 명시적으로 승인했다. 관련 gameplay ExecPlan의 design 기준으로 사용할 수 있다.
- `상태: 수정 필요`: 기존 승인이 더 이상 충분하지 않다. 다시 승인되기 전까지 gameplay ExecPlan이나 구현 기준으로 사용할 수 없다.

Gameplay ExecPlan이나 구현 기준으로 사용할 수 있는 상태는 `상태: 승인됨`뿐이다. `상태: 작성 전`, `상태: 초안`, `상태: 수정 필요`는 모두 승인 기준으로 사용할 수 없다.

인터뷰와 공유된 이해 검증을 마친 새 design 산출물은 `상태: 초안`으로 시작한다. 인터뷰 전 placeholder 파일은 `상태: 작성 전`으로 둘 수 있다. Codex는 사용자 명시 승인 없이 design 문서 상태를 `상태: 승인됨`으로 바꾸지 않는다.

## Core Beliefs 작성 절차

`docs/design/core-beliefs.md`는 Codex가 프로젝트 문서를 바탕으로 추론해서 바로 작성하지 않는다.

`core-beliefs.md`를 `상태: 초안`으로 만들기 전에 먼저 사용자 인터뷰를 진행한다. 질문은 한 번에 하나씩 하고, 각 질문에는 Codex의 추천 답안을 포함한다. 코드나 문서에서 확인할 수 있는 사실은 사용자에게 묻지 않고 직접 확인한다.

이 인터뷰는 `docs/workflows/grill-me.md`를 기준으로 한다. 각 질문에는 현재 이해, 막힌 결정, 추천 답안, 질문을 포함한다.

인터뷰 주제는 다음으로 제한한다.

- 게임 또는 실험 정체성.
- 플레이어 경험 방향.
- 초기 단계에서 하지 않을 것.
- 성공과 실패를 판단하는 기준.
- Codex가 임의로 만들면 안 되는 design 결정.

`core-beliefs.md` 인터뷰에서는 다음을 정하지 않는다.

- 첫 gameplay loop의 세부 규칙.
- 조작, 상태 변화, 성공, 실패, retry의 구체 규칙.
- 좌표, 수치, 씬 구조.
- C# 스크립트 이름.
- 구현 방식.

인터뷰 후 Codex는 목표, 성공 기준, 포함 범위, 제외 범위, 제약, 검증 방법, 위험한 가정, 남은 열린 질문, 되돌리기 비싼 결정을 요약해 공유된 이해를 검증한다.

사용자가 공유된 이해가 맞다고 확인한 뒤에만 Codex는 `docs/design/core-beliefs.md`를 `상태: 초안`으로 작성한다. `상태: 초안`은 아직 승인 상태가 아니므로, 사용자가 명시적으로 승인하기 전까지 gameplay ExecPlan이나 구현 기준으로 사용할 수 없다.

## Design 문서 생성과 갱신 분류

Codex는 gameplay 관련 요청을 처리하기 전에 기존 design 문서로 충분한지 판단한다.

- `D0`: design 문서가 필요 없음. 문서 정리, 내부 구현 세부 조정, 하네스 규칙 도입처럼 player-facing gameplay 결정을 만들지 않는 작업이다.
- `D1`: 필수 baseline 생성 또는 승인 필요. `docs/design/core-beliefs.md`가 없거나 `상태: 작성 전`, `상태: 초안`, `상태: 수정 필요`처럼 `상태: 승인됨`이 아니다.
- `D2`: 기존 design 문서 갱신 필요. 승인된 문서가 있지만 요청이 그 범위를 바꾸거나 오래된 결정을 수정한다.
- `D3`: 새 조건부 design 문서 생성 필요. 요청한 gameplay 작업을 설명하는 승인 design 문서가 아직 없다.

`D1`, `D2`, `D3`가 필요하다고 판단하면 Codex는 먼저 사용자에게 다음을 보고하고, design 문서 생성 또는 갱신 방향을 확인한다.

- 기존 문서로 충분하지 않은 이유.
- 막혀 있는 player-facing 결정.
- 필요한 조치 수준.
- 생성 또는 수정할 파일.
- 사용자에게 먼저 물어야 할 질문.

Codex는 새 design 문서를 조용히 만들지 않는다. 사용자와 범위를 확인한 뒤 `상태: 초안`으로 생성한다.

## Player-Facing Gameplay 결정 예시

다음은 design 문서에서 다뤄야 하는 player-facing gameplay 결정이다.

- 플레이어 입력 방식.
- 이동 또는 선택 규칙.
- 성공 조건.
- 실패 조건.
- Retry 또는 reset 규칙.
- UI 피드백.
- 레벨 또는 장면 목표.
- 플레이어가 볼 수 있는 상태 변화.

다음은 player-facing gameplay 결정이 아니다.

- C# 파일명.
- private 메서드 구조.
- 임시 오브젝트 이름.
- 내부 코드 분리 방식.
- git 커밋 메시지.

## Gameplay ExecPlan과 Design 기준

Player-facing gameplay ExecPlan은 승인된 `docs/design/core-beliefs.md`와 해당 작업의 플레이어 경험 또는 게임 규칙을 정의하는 승인된 design 문서 최소 1개를 함께 참조해야 한다.

`core-beliefs.md`만으로는 gameplay ExecPlan을 시작할 수 없다. `core-beliefs.md`는 프로젝트 방향 기준이고, 실제 gameplay 작업에는 해당 작업의 플레이어 경험이나 게임 규칙을 정의하는 별도 승인 문서가 필요하다.

조건부 design 문서는 작업이 필요로 할 때만 만든다. 예를 들어 gameplay loop를 구체화해야 하는 작업에서는 `gameplay-loop.md` 같은 문서가 필요할 수 있지만, 모든 게임이나 모든 단계의 필수 문서는 아니다.

## 템플릿 적용 범위

아래 템플릿은 새 design 문서 또는 기존 design 문서를 다음에 갱신할 때 적용한다.

이미 `상태: 승인됨`인 design 문서는 사용자가 명시적으로 재검토를 요청하거나, 문서 내용이 현재 작업을 뒷받침하지 못한다고 판단될 때만 갱신 대상으로 본다. 새 템플릿의 섹션명과 다르다는 이유만으로 기존 승인 design 문서를 자동으로 무효화하거나 재작성하지 않는다.

## Baseline 문서 템플릿

`docs/design/core-beliefs.md`는 다음 항목을 최소로 포함한다.

이 템플릿은 Core Beliefs 작성 절차를 거쳐 작성할 때 사용한다. 인터뷰 전 placeholder는 `상태: 작성 전`으로 두고 design 기준처럼 작성하지 않는다. 사용자 인터뷰와 공유된 이해 검증을 마친 뒤 본문을 작성할 때 `상태: 초안`으로 바꿀 수 있다.

```md
# Core Beliefs

상태: 작성 전

## 목적

이 문서가 프로젝트의 최상위 player-facing design baseline으로 어떤 기준을 제공하는지 적는다.

## 프로젝트 정체성

이 게임 또는 프로젝트가 무엇인지, 플레이어가 어떤 활동을 하는지 한두 문단으로 적는다.

## 플레이어 경험 방향

플레이어가 어떤 감정, 리듬, 난이도, 이해 가능성을 경험해야 하는지 적는다.

## 핵심 원칙

- 프로젝트가 유지해야 하는 player-facing 방향을 적는다.
- Codex와 사람이 이후 gameplay 결정을 할 때 지켜야 하는 기준을 적는다.

## 초기 단계에서 하지 않을 것

- 초기 범위에서 제외할 design 방향을 적는다.

## 성공과 실패 판단 기준

초기 플레이어 경험이 성공했다고 볼 관찰 가능한 기준과 실패 신호를 적는다.

## Codex가 임의로 확정하지 않을 것

사용자 승인 없이 Codex가 결정하면 안 되는 player-facing design 항목을 적는다.

## 아직 정하지 않을 것

후속 design 문서나 ExecPlan에서 다룰 항목을 적는다.

## 승인 체크리스트

- [ ] 사용자 인터뷰를 완료했다.
- [ ] 공유된 이해 검증에서 사용자가 의도와 맞다고 확인했다.
- [ ] 인터뷰와 검증 내용을 바탕으로 이 문서를 `상태: 초안`으로 작성했다.
- [ ] 사용자가 이 문서의 상태를 `승인됨`으로 바꾸라고 명시했다.
- [ ] 이 문서가 gameplay ExecPlan의 baseline 기준으로 충분한지 확인했다.
- [ ] 이 문서만으로 확정하지 않을 항목이 `아직 정하지 않을 것`에 분리되어 있다.
```

## 조건부 Design 문서 템플릿

조건부 design 문서는 다음 항목을 최소로 포함한다.

```md
# 문서 제목

상태: 작성 전

## 목적

이 문서가 어떤 player-facing gameplay 결정 기준을 제공하는지 설명한다.

## 적용 범위

이 문서가 다루는 플레이어 경험, 게임 규칙, 화면 상태를 적는다.

## 제외 범위

이 문서가 일부러 결정하지 않는 항목을 적는다.

## 플레이어 경험 기준

플레이어가 무엇을 이해하고, 느끼고, 판단할 수 있어야 하는지 적는다.

## 결정 사항

- 승인할 player-facing gameplay 결정을 구체적으로 적는다.

## 검증 기준

이 design 기준이 구현됐는지 Play Mode, 수동 체크, 테스트, 문서 검토 등으로 어떻게 확인할지 적는다.

## 열린 질문

- 아직 승인되지 않은 질문을 적는다. 없으면 `없음`이라고 적는다.

## Codex 금지 사항

이 문서를 근거로 Codex가 임의로 확장하거나 확정하면 안 되는 항목을 적는다.

## 승인 체크리스트

- [ ] 사용자 인터뷰를 완료했다.
- [ ] 공유된 이해 검증에서 사용자가 의도와 맞다고 확인했다.
- [ ] 인터뷰와 검증 내용을 바탕으로 이 문서를 `상태: 초안`으로 작성했다.
- [ ] 사용자가 이 문서의 상태를 `승인됨`으로 바꾸라고 명시했다.
- [ ] 관련 gameplay ExecPlan이 새 규칙을 발명하지 않고 이 문서를 참조한다.
```
