5분 목표 브리핑

Tiny Garden Restore는 게임보다 하네스 v1 검증이 먼저다

`codex-harness-tgr-v1`은 DBZ에서 검증한 최소 Codex 하네스를 두 번째 Unity 프로젝트에 이식하고, 상태 변화, 해금, 데이터 기반 진행, 검증, 세션 인수인계를 더 작은 게임 실험 안에서 확인하는 프로젝트다.

핵심 판단: TGR은 “조금 더 게임다운 DBZ 후속작”이 아니라, 반복 가능한 Unity/Codex 개발 방식을 남기기 위한 하네스 검증 프로젝트다.

5분 읽기 순서

  1. 한 문장 목표를 먼저 잡는다.
  2. DBZ와 TGR의 검증 축 차이를 본다.
  3. 5개 실험 질문을 확인한다.
  4. 단계 0-5 흐름을 훑는다.
  5. 성공 기준과 보류 항목만 남긴다.

DBZ에서 이미 확인한 것

검증 축

단일 플레이 루프

작은 Unity 2D 퍼즐 게임을 Codex 기반 흐름으로 끝까지 만들 수 있는지 확인했다.

중심 흐름

Move, Pickup, Deliver, Retry

움직임, 승패, 재시작 같은 명확한 규칙을 구현하고 검증했다.

TGR에서 새로 확인할 것

검증 축

상태와 진행

오브젝트 상태 변화, 해금 조건, 데이터 기반 진행을 안정적으로 다룰 수 있는지 확인한다.

중심 흐름

Select, Restore, Unlock, Reset

정원 오브젝트가 복구되고 다음 상태나 다음 오브젝트로 이어지는 흐름을 만든다.

이번 프로젝트가 답해야 하는 5개 질문

하네스 재사용 DBZ 문서 하네스를 다른 Unity 프로젝트에 옮겨도 작동하는가?
규칙 생성 방지 Codex가 사용자 인터뷰 없이 게임 규칙을 임의로 만들지 않게 막는가?
Unity 선택 고정 셋업, 폴더, 데이터, 검증 기준을 v1 문서로 고정할 수 있는가?
상태 기반 구현 코드 구현을 넘어 상태 변화, 해금, 데이터 진행을 안정적으로 구현하는가?
다음 프로젝트 복사 다음 Unity 실험에도 복사 가능한 최소 하네스 세트를 만들 수 있는가?

추천 흐름: 단계 0-5

단계 0

Bootstrap & Harness Reuse Check

DBZ 하네스를 TGR에 이식하고 DBZ 전용 문구가 남아 있는지 검증한다.

단계 1

Core Beliefs & Game Loop Interview

구현 전에 TGR이 어떤 게임인지 사용자 인터뷰로 확정한다.

단계 2

Unity Setup + Data Model

Unity 구조와 GardenObject 데이터 모델을 먼저 고정한다.

단계 3

First Playable Restore Loop

오브젝트 1개를 선택하고 복구하는 최소 플레이 루프를 만든다.

단계 4

Unlock / Visual State / Verification Loop

2-3개 오브젝트로 상태 변화, 해금, 검증 문서 연결을 확인한다.

단계 5

Handoff + Harness v1 Retrospective

문서만 읽고 이어받을 수 있는지 확인하고 v1 평가를 남긴다.

TGR에서 DBZ보다 한 단계 올릴 산출물

Unity 셋업 문서

Unity 버전, 2D/3D, URP, 씬, 입력, UI, 데이터, 테스트 선택을 고정한다.

GardenObject 데이터

Locked, Available, Restored 같은 상태와 해금 조건을 데이터로 표현한다.

작은 아트 방향성

따뜻한 정원 복구, 파스텔톤, 임시 표식 허용 같은 화면 기준을 잡는다.

검증 체크리스트

Play Mode와 데이터 상태 검증을 분리해 복구, 해금, Reset을 관찰한다.

하네스 v1 회고

무엇을 재사용할지, 무엇을 보류할지, skill로 뺄 후보가 생겼는지 정리한다.

성공 기준은 3단계로 본다

최소 성공

  • DBZ 하네스가 TGR에 이식됨
  • DBZ 전용 문구가 제거됨
  • core-beliefs가 작성 전 상태로 시작함
  • Codex가 인터뷰 없이 구현을 시작하지 않음

실질 성공

  • core-beliefs와 gameplay loop가 승인됨
  • Unity project setup 문서가 생김
  • GardenObject 상태 모델이 생김
  • Play Mode에서 오브젝트 1개 이상을 복구함

완전 성공

  • 오브젝트 2-3개가 상태 변화와 해금 흐름을 가짐
  • 데이터/상태 검증 기준이 있음
  • 새 Codex 세션이 문서만 읽고 이어받을 수 있음
  • 다음 프로젝트용 template 후보와 보류 도구가 구분됨

이번 프로젝트에서 보류할 것

TGR의 목적은 하네스 v1 검증이다. 기술 스택 확장은 반복 마찰이 실제로 확인된 뒤 검토한다.

  • Unity MCP 도입
  • hooks 도입
  • subagent 도입
  • custom skill 설치
  • 복잡한 Clean Architecture
  • VContainer/R3/UniTask 도입
  • 세이브/로드 본격 구현
  • 상점/재화/인벤토리 확장
  • 3D 전환
  • 외부 에셋 대량 사용

지금 바로 할 일

현재 다음 작업은 구현이 아니라 프로젝트 목표 고정이다. 저장소가 하네스를 제대로 초기화했는지 점검하고, 남은 DBZ 전용 내용과 TGR 장기 결정을 분리한 뒤, 단계 1에서 물어야 할 인터뷰 질문을 준비한다. 

README.md, AGENTS.md, PLANS.md, docs/current-state.md,
docs/decisions.md, docs/design/README.md,
docs/design/core-beliefs.md, exec-plans/000-bootstrap.md를 읽어라.

목적: 두 번째 프로젝트의 목표를 확정하기 전 상태를 점검한다.

1. DBZ 하네스를 올바르게 초기화했는지 점검한다.
2. DBZ 전용 문구, 결정, gameplay 기준이 남았는지 찾는다.
3. current-state가 TGR 상태를 정확히 설명하는지 확인한다.
4. 장기 하네스 결정과 제거할 DBZ 전용 결정을 구분한다.
5. core-beliefs가 상태: 작성 전인지 확인한다.
6. 아직 gameplay 구현, Unity 씬 구현, 새 코드 작성은 하지 않는다.
