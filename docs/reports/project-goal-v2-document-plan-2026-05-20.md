# project-goal-v2 작성 준비 메모

## 목적

이 문서는 v1 완료 후 `docs/project-goal-v2.md`를 만들 때 참고할 작성 기준과 검토 후보를 보존한다.

현재 시점의 목적은 v2를 시작하거나 v2 실행 계획을 확정하는 것이 아니다. v1 playable loop와 경량 하네스 평가가 끝난 뒤, 실제 결과를 바탕으로 v2 기준 문서를 작성하기 위한 메모를 남기는 것이다.

이 문서에 적힌 항목은 v2 필수 범위가 아니라, v1 완료 후 선별할 검토 후보이다. 실제 `docs/project-goal-v2.md`에는 v1 결과에 근거해 필요한 항목만 포함한다.

## 현재 하지 않을 것

현재는 다음 작업을 하지 않는다.

- `docs/project-goal-v2.md` 생성
- `docs/ai-mistakes.md` 생성
- `AGENTS.md`에 AI 실수 장부 섹션 추가
- `memory-bank/` 또는 별도 memory 체계 생성
- `docs/project-goal.md`의 v1 게임 규칙 수정
- 이미지 교체
- 게임 규칙 변환 기획 확정
- v2 체크리스트 생성

## 나중에 검토할 수 있는 후보 주제

아래 항목은 v2 필수 범위가 아니라, v1 완료 후 선별할 후보 주제다. 실제 `docs/project-goal-v2.md`에는 v1 결과에 근거해 필요한 항목만 포함한다.

- v1과 v2의 경계
- v2 시작 조건
- 이미지 교체 실험의 목적과 범위
- AI 반복 실수 기록 문서의 필요성
- `docs/ai-mistakes.md`를 만들 경우의 명명, 위치, 기록 기준
- v1 playable loop가 검증 대상으로 충분했는지 평가
- 게임 개선이 필요할 경우의 최소 규칙 보정 범위
- 게임 규칙 변환 실험의 목적과 범위
- `기획 -> 아트 -> 개발` 순서
- v1/v2 비교 평가 방식

## `docs/ai-mistakes.md` 검토 기준

v2에서 AI 반복 실수 기록 문서를 만든다면, 이 문서는 Memory Bank가 아니라 경량 재발 방지 장부로 취급한다.

예상 위치와 이름은 다음을 우선 검토한다.

```text
docs/ai-mistakes.md
```

기록 대상은 다음처럼 실제 재작업이나 품질 저하로 이어진 구체적 실수로 제한한다.

- 사용자가 명시적으로 잘못됐다고 지적한 실수
- Unity Play Mode, Console, 테스트, 빌드에서 확인된 실수
- 게임 방향과 다른 기능을 만든 범위 이탈
- 승인되지 않은 설계, 구조, 패키지, 시스템 추가
- 다음 세션에서도 같은 교정을 반복해야 할 가능성이 높은 실수

기록하지 않을 것은 다음이다.

- 단순 오타
- 한 번만 발생한 사소한 착오
- 취향 수준의 선호
- 아직 검증되지 않은 추측
- 너무 일반적인 조언

운영 방식은 "실수 발견 -> 현재 작업 보고 또는 ExecPlan에 먼저 기록 -> 반복 가능성이 있으면 `docs/ai-mistakes.md`로 승격"을 기본값으로 검토한다.

단, 반복 실수가 1~2개 수준이고 현재 작업 보고나 ExecPlan 회고에 충분히 남길 수 있다면 별도 `docs/ai-mistakes.md`를 만들지 않는다.

`AGENTS.md`에는 실제 실수 사례를 직접 누적하지 않는다. v2에서 `docs/ai-mistakes.md`를 만들기로 결정한 경우에만, 구현 전 확인과 기록 조건을 안내하는 짧은 섹션을 추가할지 검토한다.

## 게임 개선 검토 기준

v1의 단순한 규칙은 실패로 보지 않는다. v1의 목적은 큰 게임 시스템이 아니라 작은 playable loop와 경량 하네스 검증이기 때문이다.

다만 M5 완료 후 실제 플레이가 "세 대상에 다가가 E를 한 번씩 누르고 끝나는 흐름"에 머물러 검증할 것이 너무 적었다고 판단되면, v2 문서에서 최소한의 게임 개선을 검토한다.

기본 원칙은 다음이다.

```text
규칙은 보정할 수 있지만, 새 시스템은 추가하지 않는다.
```

우선 검토할 개선 후보는 다음이다.

- 복구 순서는 자유로 유지한다.
- 완료 조건은 순서가 아니라 `복구된 대상 수가 3개인지`로 판단한다.
- 복구 수에 따라 정원 회복도를 보여준다.
  - 0/3: Neglected Garden
  - 1/3: Getting Better
  - 2/3: Almost Restored
  - 3/3: Garden Restored
- 대상 근처에서 `E: Restore` 같은 최소 상호작용 피드백을 보여줄지 검토한다.
- 복구 후 상태 변화가 충분히 읽히는지 확인하고, 필요하면 짧은 시각 피드백을 검토한다.

이 개선 후보들은 점수, 성장, 경제, 장기 진행 시스템이 아니라, 현재 정원이 회복되고 있음을 플레이어가 읽게 만드는 최소 피드백으로 다룬다.

게임 개선 후보는 전부 적용하는 것이 아니라, v1 평가 결과 가장 작은 보정으로 효과를 볼 수 있는 항목만 선택한다.

우선순위는 다음과 같이 검토한다.

1. 복구 진행도 표시
2. `E: Restore` 상호작용 피드백
3. 짧은 복구 후 시각 피드백

v2 문서에서 제외 후보로 우선 검토할 항목은 다음이다.

- 물뿌리개, 삽, 빗자루 같은 도구 선택 시스템
- 대상별 도구 매칭
- 인벤토리
- 복구 시간 게이지
- 스태미나
- 제한 시간
- 점수, 별점, 콤보
- 하루/밤 루프
- 복구 순서 퍼즐
- 랜덤 배치
- 여러 정원 또는 여러 스테이지

위 항목은 매력적인 확장일 수 있지만, v2 문서를 만들 때 별도 승인 없이 기본 범위로 넣지 않는다.

## 작성 시점

`docs/project-goal-v2.md`는 다음 조건을 확인한 뒤 작성한다.

1. M5 playable loop가 완료되어 있다.
2. M6 경량 하네스 평가가 끝나 있다.

위 조건을 충족하기 전에는 v2 문서를 실행 기준으로 만들지 않는다.

v2에서 이미지 교체나 게임 규칙 보정을 진행하더라도, 작업 순서는 `기획 확정 -> 아트 기준 확정 -> 개발`을 기본값으로 검토한다. 개발을 먼저 시작하지 않는다.

## 작성 전 재검토 항목

v2 문서 작성 전에는 다음 질문을 먼저 확인한다.

- 실제 v1 결과가 무엇인가?
- 어떤 AI 실수가 반복됐는가?
- 반복 실수가 현재 작업 보고에만 남아도 충분한가, 별도 `docs/ai-mistakes.md`로 승격할 필요가 있는가?
- 이미지 교체가 필요한 대상은 무엇인가?
- 현재 규칙에서 무엇을 변환할 가치가 있는가?
- v1의 세 복구 대상이 플레이 규칙상 모두 같은 행동으로 느껴졌는가?
- `복구 순서 자유`, `정원 회복도`, `E: Restore` 피드백만으로 v2 검증 범위가 충분히 늘어나는가?
- 추가하려는 개선이 규칙 보정인지, 인벤토리/도구/점수 같은 새 시스템인지?

## 작성 기준

나중에 작성할 `docs/project-goal-v2.md`는 v1 결과에 근거해야 한다. 현재 추정만으로 v2 범위, 체크리스트, 구현 순서를 확정하지 않는다.

이 문서는 `docs/project-goal-v2.md` 자체가 아니라, 그 문서를 나중에 만들기 위한 작성 계획이다.
