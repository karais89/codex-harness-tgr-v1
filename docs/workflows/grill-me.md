# Grill-Me Discovery Protocol

## 역할

이 문서는 `codex-harness-tgr-v1`에서 creative, art, design, ExecPlan 작업을 시작하기 전에 사용하는 공통 인터뷰 프로토콜이다.

이 문서는 custom skill이 아니다. 아직 이 저장소에는 MCP, custom skill, hook, subagent를 추가하지 않는다. 이 문서는 repo-local workflow protocol이며, 실제 반복 사용으로 안정성이 확인되기 전까지 skill로 승격하지 않는다.

## 출처와 라이선스

이 프로토콜은 `mattpocock/skills`의 `grill-me` skill을 기준으로 삼는다.

- 원문: https://github.com/mattpocock/skills/blob/main/skills/productivity/grill-me/SKILL.md
- 라이선스: MIT License
- 저작권 고지: Copyright (c) 2026 Matt Pocock

원문은 짧은 행동 규칙이다. 이 저장소에서는 다음 원문 기준을 유지한다.

- 공유된 이해에 도달할 때까지 인터뷰한다.
- design tree의 각 분기를 의존성 순서에 따라 하나씩 해결한다.
- 질문은 한 번에 하나씩 한다.
- 각 질문에는 Codex의 추천 답안을 포함한다.
- 코드베이스나 문서에서 답을 확인할 수 있으면 사용자에게 묻지 않고 직접 확인한다.

아래의 적용 대상, 종료 조건, 승인 게이트, 영역별 decision branch는 이 저장소의 Unity 하네스 검증을 위한 로컬 확장이다. 로컬 확장은 원문의 질문 방식보다 우선하지 않는다.

`deep-interview`는 이번 프로토콜의 기준으로 채택하지 않는다. 이 프로젝트의 목적은 빠른 요구사항 정리가 아니라, 기획, 아트, design, 실행 계획의 암묵적 결정을 충분히 캐내는 것이다.

## 적용 대상

Codex는 다음 작업을 시작하기 전에 이 문서를 읽고 적용한다.

- `creative`: 아이디어 발산, 후보 압축, concept brief 작성
- `art`: 아트 방향 탐색, 이미지 프롬프트 작성, 생성 이미지 후보 판단
- `design`: Core Beliefs, 조건부 design 문서, player-facing 기준 작성 또는 갱신
- `ExecPlan`: 승인된 기준을 구현과 검증 순서로 옮기는 계획 작성

분석, 검토, 단순 오타 수정, 작은 서식 수정처럼 사용자가 명확히 파일 변경 없는 작업이나 아주 작은 수정을 요청한 경우에는 이 프로토콜을 강제하지 않는다.

## 원칙

Codex는 사용자가 피로해 보인다는 이유만으로 discovery를 조기 종료하지 않는다.

중요한 decision branch, 암묵적 가정, 제외 범위, 판단 기준, 승인 조건이 남아 있으면 계속 질문한다.

질문은 한 번에 하나만 한다.

각 질문에는 다음을 포함한다.

- 현재 이해: Codex가 지금까지 이해한 내용을 한두 문장으로 요약한다.
- 막힌 결정: 지금 풀어야 하는 가장 중요한 decision branch를 밝힌다.
- 질문: 사용자가 답해야 하는 한 가지 질문만 묻는다.
- 추천 답안: Codex가 보기에는 어떤 선택이 가장 안전한지 제안한다.

코드, 문서, git 상태, 기존 ExecPlan, 기존 design 문서에서 확인할 수 있는 사실은 사용자에게 묻지 않는다. Codex가 직접 확인하고, 확인한 사실을 질문 맥락에 반영한다.

사용자가 `기본값`, `기본 제안`, `추천대로`처럼 답하면 Codex의 추천 답안을 채택할 수 있다. 단, 되돌리기 비싼 결정, player-facing 기준, 구현 승인, design 승인은 명시 질문과 명시 확인을 다시 거친다.

사용자가 `멈춰`라고 말하면 즉시 중단한다. 중단 전까지의 결정사항을 파일에 쓰지 않는다. 사용자가 요청하면 그때 요약만 제공한다.

## 진행 방식

1. 요청을 분류한다.
   - creative, art, design, ExecPlan 중 어디에 속하는지 판단한다.
   - 분류가 애매하면 파일을 수정하지 않고 먼저 질문한다.

2. 저장소에서 확인 가능한 사실을 먼저 확인한다.
   - `README.md`, `AGENTS.md`, `docs/current-state.md`, `docs/decisions.md`를 읽는다.
   - 영역별 README를 읽는다.
   - 관련 ExecPlan이나 design 문서가 있으면 읽는다.

3. decision tree의 첫 번째 막힌 분기를 고른다.
   - 가장 되돌리기 비싼 결정부터 묻는다.
   - 후속 질문의 답을 바꿀 수 있는 상위 결정을 먼저 묻는다.

4. 한 번에 하나씩 질문한다.
   - 질문마다 현재 이해, 막힌 결정, 질문, 추천 답안을 제시한다.
   - 사용자의 답변 뒤에는 결정된 내용과 남은 막힌 분기를 짧게 갱신한다.
   - 아직 사용자가 답하지 않은 내용은 `결정`으로 쓰지 않고 `현재 가정`이나 `추천 답안`으로 표시한다.

5. 공유된 이해를 검증한다.
   - 목표
   - 성공 기준
   - 포함 범위
   - 제외 범위
   - 제약
   - 검증 방법
   - 위험한 가정
   - 남은 열린 질문
   - 되돌리기 비싼 결정

6. 다음 단계 승인을 별도 질문으로 확인한다.
   - creative/art 문서에 기록할지 확인한다.
   - design 초안으로 넘길지 확인한다.
   - ExecPlan을 만들지 확인한다.
   - 구현을 시작할지 확인한다.

짧은 긍정 답변은 직전 명시 질문에만 승인으로 본다. 일반적인 동의 표현을 다음 단계 승인으로 확대 해석하지 않는다.

## 종료 조건

Codex는 다음 중 하나가 충족될 때만 discovery를 종료한다.

- 사용자가 명시적으로 중단을 요청한다.
- 사용자가 명시적으로 지금까지의 내용을 문서화하라고 요청한다.
- 사용자가 공유된 이해 검증이 맞다고 확인하고, 다음 단계 승인 질문에 답한다.
- 현재 작업의 중요한 decision branch가 모두 닫혔고, 열린 질문이 의도적으로 보류되었다.

Codex는 `질문이 많아 보인다`, `대충 충분해 보인다`, `이 정도면 구현 가능해 보인다`는 이유로 종료하지 않는다.

## Creative 적용

Creative 작업에서 Codex는 최소한 다음 decision branch를 확인한다.

- 이 게임 또는 실험의 정체성
- 플레이어가 느껴야 하는 감정과 리듬
- 초기 단계에서 하지 않을 것
- 브레인스토밍을 넓힐 기준
- 후보를 선택, 보류, 폐기할 기준
- concept brief로 넘길 수 있는 최소 합의
- design 문서로 넘기면 안 되는 미확정 아이디어

Creative 산출물은 확정 구현 사양이 아니다. `docs/creative`에 기록된 내용이 player-facing 규칙, 목표, 조작, 피드백, 화면 상태, 구현 기준으로 바뀌려면 별도 design 절차를 거친다.

## Art 적용

Art 작업에서 Codex는 최소한 다음 decision branch를 확인한다.

- 원하는 분위기와 피해야 할 분위기
- 색감, 명도, 대비, 형태 언어
- 레퍼런스나 이미지 후보가 art reference인지 player-facing 기준인지
- 이미지 프롬프트가 무엇을 검증하려는지
- 생성 이미지를 저장할지 버릴지
- 저장한 이미지를 채택, 보류, 폐기 중 어디에 둘지
- Unity 에셋 사용 승인 여부

생성 이미지나 프롬프트는 그 자체로 gameplay 사양이나 Unity 에셋 사용 승인이 아니다.

## Design 적용

Design 작업에서 Codex는 최소한 다음 decision branch를 확인한다.

- 이 문서가 제공해야 하는 player-facing 기준
- 적용 범위와 제외 범위
- 플레이어 경험 기준
- 승인할 결정과 아직 정하지 않을 결정
- 구현 기준으로 삼아도 되는 검증 방법
- Codex가 임의로 확장하면 안 되는 항목

Design 문서는 creative/art 문서보다 더 엄격한 기준을 가진다. `상태: 초안`은 구현 기준이 아니며, 사용자가 명시 승인한 `상태: 승인됨` 문서만 gameplay ExecPlan이나 구현 기준으로 사용할 수 있다.

## ExecPlan 적용

ExecPlan 작업에서 Codex는 최소한 다음 decision branch를 확인한다.

- 어떤 승인 기준을 구현 또는 검증 대상으로 삼는지
- 포함 범위와 제외 범위
- 구현 순서
- 검증 방법
- 문서 갱신 범위
- 완료 기준
- 구현 승인 전에는 하지 않을 작업

ExecPlan은 새로운 player-facing 규칙을 발명하지 않는다. 승인된 design 또는 workflow 기준을 실행 순서와 검증 절차로 옮기는 문서다.
