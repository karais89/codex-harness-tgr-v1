# Codex 실행 계획 (ExecPlans)

이 파일은 이 저장소에서 ExecPlan을 작성하고 유지하는 방법을 정의합니다.

ExecPlan은 자기완결적인 실행 문서입니다. 새로운 Codex 세션이나 사람 개발자가 이전 채팅 기록에 의존하지 않고, 저장소와 ExecPlan만 읽고도 작업을 완료할 수 있어야 합니다.

ExecPlan은 살아 있는 문서입니다. 작업이 진행되거나, 새 사실을 발견하거나, 결정이 바뀔 때마다 갱신해야 합니다.

## 살아 있는 계획과 설계 결정

모든 ExecPlan은 필수 섹션을 포함하고 작업 중 계속 갱신해야 합니다.

프로젝트 상태, 현재 단계, 활성 계획, 다음 단계, 알려진 제한 사항이 바뀌면 `docs/current-state.md`를 갱신합니다.

오래 유지해야 하는 프로젝트 결정이 생기면 ExecPlan의 `결정 기록`에 남기는 것과 별도로 `docs/decisions.md`를 갱신합니다.

ExecPlan의 `결정 기록`은 해당 계획 안에서 발생한 판단과 변경 이유를 기록합니다.

`docs/decisions.md`는 특정 계획을 넘어 계속 따라야 하는 프로젝트 결정을 기록합니다.

## ExecPlan 저장 위치와 파일 이름

ExecPlan은 저장소 루트의 `exec-plans/` 아래에 저장합니다.

파일 이름은 다음 형식을 사용합니다.

`NNN-short-kebab-title.md`

예시:

- `exec-plans/000-bootstrap.md`
- `exec-plans/001-first-playable-loop.md`
- `exec-plans/002-play-mode-verification.md`

## ExecPlan을 사용하는 경우

사소하지 않은 작업에는 ExecPlan을 사용합니다.

사소하지 않은 작업의 예시는 다음과 같습니다.

- 새로운 게임플레이 동작
- Unity 씬 변경
- 아키텍처 또는 폴더 구조 변경
- 워크플로우 변경
- 테스트 또는 검증 방식 변경
- 동작이나 구조를 바꾸는 리팩터링
- 패키지 추가 또는 제거
- 프로젝트 관례 변경

다음 작업에는 ExecPlan이 필요하지 않습니다.

- 오타 수정
- 작은 서식 수정
- 아주 작은 README 변경
- 동작이나 워크플로우에 영향을 주지 않는 주석

## Design Dependency Check

Player-facing gameplay ExecPlan을 작성하거나 갱신하기 전에 Codex는 승인된 Design 기준을 확인합니다.

승인된 Design 기준은 `상태: 승인됨`인 실제 design 산출물(`docs/design/*.md`)만 될 수 있습니다. `상태: 작성 전`, `상태: 초안`, `상태: 수정 필요` 문서는 gameplay ExecPlan 기준으로 사용할 수 없습니다.

확인 대상은 다음과 같습니다.

1. `docs/design/README.md`가 존재한다.
2. `docs/design/core-beliefs.md`가 존재하고 `상태: 승인됨`이다.
3. 해당 작업의 플레이어 경험 또는 게임 규칙을 정의하는 승인된 `docs/design/*.md` 문서가 최소 1개 있다.

`docs/design/README.md`는 design 문서 작성 규약이므로 존재 확인 대상입니다. 승인 상태 확인 대상은 실제 design 산출물인 `docs/design/*.md` 파일입니다.

`docs/design/core-beliefs.md`가 `상태: 작성 전`이면 파일이 존재해도 design 산출물이 아닙니다. 이 경우 Codex는 gameplay ExecPlan 작성을 멈추고 Core Beliefs 인터뷰와 공유된 이해 검증이 먼저 필요하다고 알립니다.

`core-beliefs.md`만으로는 gameplay ExecPlan을 시작할 수 없습니다. `core-beliefs.md`는 프로젝트 방향 기준이고, gameplay ExecPlan은 해당 작업의 플레이어 경험 또는 게임 규칙을 정의하는 별도 승인 design 문서를 함께 참조해야 합니다.

관련 승인 design 문서가 없으면 Codex는 ExecPlan 작성을 멈추고 design 확인을 요청합니다. 이때 기존 문서로 충분하지 않은 이유, 막혀 있는 player-facing 결정, 필요한 조치 수준, 생성 또는 수정할 파일, 사용자에게 먼저 물어야 할 질문을 보고합니다.

Gameplay ExecPlan은 새로운 player-facing 규칙을 발명하지 않습니다. 승인된 design 문서를 구현 방법과 검증 절차로 옮기는 역할만 합니다.

하네스, 문서 구조, workflow 규칙 도입 작업에는 이 체크를 적용하지 않습니다. 단, 그런 작업 중 player-facing gameplay 규칙을 정의하기 시작하면 Design Dependency Check로 전환합니다.

Gameplay ExecPlan에는 다음 섹션을 포함합니다.

```md
## 승인된 Design 기준

- `docs/design/core-beliefs.md` 상태: 승인됨
- `docs/design/<relevant-design-doc>.md` 상태: 승인됨
```

## ExecPlan 전 인터뷰와 생성 게이트

사소하지 않은 작업은 ExecPlan 작성 전에 사용자 의도와 작업 범위를 확인합니다.

ExecPlan 전 인터뷰는 `docs/workflows/grill-me.md`의 `Grill-Me Discovery Protocol`을 기준으로 합니다. 이 인터뷰는 질문을 줄여 빠르게 끝내는 절차가 아니라, 구현 순서와 검증 방법에 영향을 주는 decision tree가 닫힐 때까지 공유된 이해를 만드는 절차입니다.

기본 흐름은 다음과 같습니다.

1. 논의
2. `grill-me` 방식 인터뷰
3. 공유된 이해 검증
4. ExecPlan 생성 승인 확인
5. ExecPlan 생성
6. Codex 자체 리뷰
7. 사용자 검토용 요약
8. 구현 승인 확인
9. 구현
10. 검증과 문서 갱신

인터뷰는 한 번에 하나의 중요한 결정 분기만 묻습니다. 각 질문에는 현재 이해, 막힌 결정, 질문, Codex의 추천 답안을 포함합니다. 코드나 문서에서 확인할 수 있는 사실은 사용자에게 묻지 않고 직접 확인합니다.

공유된 이해 검증에서는 다음을 요약합니다.

- 목표
- 성공 기준
- 포함 범위
- 제외 범위
- 제약
- 검증 방법
- 위험한 가정
- 남은 열린 질문
- 되돌리기 비싼 결정

공유된 이해 검증이 끝난 뒤 Codex는 다음 질문으로 ExecPlan 생성 승인을 확인합니다.

```text
이 내용으로 ExecPlan을 만들까요?
```

이 질문 직후의 짧은 긍정 답변은 ExecPlan 생성 승인으로 봅니다. 공유된 이해 검증 없이 나온 일반적인 동의 표현은 ExecPlan 생성 승인으로 보지 않습니다.

새 ExecPlan은 기본적으로 `구현 승인 대기` 상태로 시작합니다. 진행 상황 섹션에는 현재 상태가 드러나야 합니다.

## ExecPlan 생성 후 자체 리뷰

ExecPlan을 생성한 직후 Codex는 사용자가 검토하기 전에 자체 리뷰를 수행합니다.

자체 리뷰에서는 다음을 확인합니다.

- 인터뷰 합의와 ExecPlan 내용이 일치하는가
- 목표, 범위, 제외 범위, 검증 방법이 빠지지 않았는가
- 구현 승인 전인데 구현 작업이 포함되어 있지 않은가
- 범위가 조용히 확장되지 않았는가
- 위험한 가정이나 열린 질문이 표시되어 있는가
- 완료 기준이 관찰 가능한가

문제가 발견되면 ExecPlan을 수정한 뒤 다시 자체 리뷰합니다.

자체 리뷰 후 Codex는 사용자 검토용 요약을 제공합니다. 요약에는 목표, 포함 범위, 제외 범위, 검증 방법, 현재 상태, 사용자가 특히 확인해야 할 결정 3개 이하를 포함합니다.

그 뒤 Codex는 다음 질문으로 구현 승인을 확인합니다.

```text
이 ExecPlan대로 구현을 시작할까요?
```

이 질문 직후의 짧은 긍정 답변은 구현 승인으로 봅니다. 검토 중 나온 일반적인 동의 표현은 구현 승인으로 보지 않습니다.

별도 `game-brief` 또는 `game-spec` 문서는 기본 산출물로 만들지 않습니다. 검토용 HTML이나 ExecPlan 요약 문서 같은 보조 산출물은 계획이 복잡하거나 사용자가 요청하고 동의한 경우에만 생성합니다.

## 필수 읽기 순서

ExecPlan을 작성하거나 수정하기 전에 다음을 읽습니다.

1. `README.md`
2. `AGENTS.md`
3. `PLANS.md`
4. `docs/current-state.md`가 있으면 읽기
5. `docs/decisions.md`가 있으면 읽기
6. `docs/workflows/grill-me.md`가 있으면 읽기
7. 관련 ExecPlan이 있으면 읽기

## 필수 섹션

모든 ExecPlan은 다음 섹션을 포함해야 합니다.

### 1. 목적

목표를 쉬운 말로 설명합니다.

계획이 완료된 뒤 지금은 할 수 없지만 새로 가능해지는 일을 설명합니다.

관찰 가능한 결과가 명확하지 않다면 “시스템 개선”이나 “코드 정리” 같은 모호한 목표는 피합니다.

### 2. 진행 상황

체크박스를 사용합니다.

작업이 시작되거나, 멈추거나, 방향이 바뀌거나, 완료될 때마다 이 섹션을 갱신합니다.

사소하지 않은 작업의 새 ExecPlan은 기본적으로 `구현 승인 대기` 상태로 시작하고, 사용자 구현 승인 후 `진행 중`으로 갱신합니다.

예시:

- 상태: 구현 승인 대기
- [x] Unity 프로젝트 생성.
- [x] `README.md` 추가.
- [ ] Unity 프로젝트가 열리는지 확인.
- [ ] 최종 bootstrap 단계 상태를 GitHub에 push.

### 3. 맥락

프로젝트의 현재 상태를 설명합니다.

관련 파일, 폴더, 씬, 스크립트, 문서의 정확한 이름을 포함합니다.

독자가 이전 채팅 기록을 기억한다고 가정하지 않습니다.

### 4. 계획

작업을 작고 순서 있는 단계로 설명합니다.

각 단계는 Codex가 추측 없이 실행할 수 있을 만큼 구체적이어야 합니다.

넓은 수평적 준비 작업보다 작고 검증 가능한 수직 단위를 선호합니다.

### 5. 검증

작업 완료를 어떻게 증명할지 설명합니다.

검증은 관찰 가능해야 합니다.

다음 중 하나 이상을 사용합니다.

- Unity Editor 관찰
- Play Mode 동작
- Edit Mode 테스트
- 명령 출력
- 파일 존재 확인
- `git status` 확인
- 문서화된 수동 검증

코드나 파일이 변경되었다는 이유만으로 계획을 완료 처리하지 않습니다.

### 6. 결정 기록

중요한 결정은 다음 형식으로 기록합니다.

- 결정:
- 근거:
- 날짜:

범위가 바뀌거나, 트레이드오프가 발생하거나, 복잡한 방법 대신 더 단순한 방법을 선택한 경우 이 섹션을 사용합니다.

### 7. 예상 밖 발견

예상하지 못한 사실을 기록합니다.

예시:

- Unity가 예상과 다르게 파일을 생성했다.
- 패키지 또는 API를 사용할 수 없었다.
- 더 단순한 구현 경로를 발견했다.
- 원래 계획이 너무 컸다.

### 8. 회고

계획이 끝나면 다음을 요약합니다.

- 완료한 것
- 완료하지 못한 것
- 배운 것
- 다음에 해야 할 것
- 다음 계획을 시작할 준비가 되었는지 여부
